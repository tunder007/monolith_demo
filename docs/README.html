Softeneers Framework — Documentation

The docs are organized in three layers, from "just get it running" to "every package explained as standalone software." Read as deep as you need.

The three layers

LayerForStart here
0 — QuickstartSpin up and run a new project in any combination. Nothing else.layer-0-quickstart.md
1 — Packages & combinationsWhat the @softeneers/* packages do and how each template/toggle combination uses them.layer-1-packages.md
2 — ReferenceEach package as a separate piece of software: full API, usage, internals.layer-2-reference/

Most people start at Layer 0, dip into Layer 1 to understand the building blocks, and consult Layer 2 when integrating a specific package.

Supporting docs

These cut across the layers:

  • ARCHITECTURE.md — the monorepo layout and the

package-vs-template distinction.

  • DECISIONS.md — authoritative decisions / tiebreakers. **When

two docs seem to conflict, this file wins.**

  • CLI-SPEC.md — the full create-softeneers-app contract

(flags, prompts, the toggle/fragment engine).

  • ROADMAP.md — sprints and current status.
  • PUBLISHING.md — how the packages get to npm.

Related at the repo root: ../README.md (overview), ../CONTRIBUTING.md (setup), ../CLAUDE.md (agent entry contract).

Conventions

  • .md files are the AI-optimized source of truth; their .html companions are

the generated human view (see DECISIONS.md D-07). Each doc has a committed .html beside it, with a browsable index.html at the repo root, generated by apps/docs (npm run build) — never hand-edit the HTML.

  • Status claims live in ROADMAP.md; other docs link to it rather than

duplicating status.

  • Every relative link must resolve (checked in CI via the docs-link test).

The deployed, public marketing+docs site is live at <https://softeneers-landing.vercel.app> (source in apps/landing).
